## 🔴 ERRORS / FAULTS NOT IN EXISTING TODO LISTS

### 1. `emitParseError` Routes User Syntax Errors Through `emitInternalError`

```cpp
void BisonParseListener::emitParseError( const silly::BisonParser::location_type& bLoc, const std::string& msg )
{
    mlir::Location loc = getLocation( bLoc );
    emitInternalError( loc, __FILE__, __LINE__, __func__, msg, currentFuncName );
}
```

This stamps user syntax errors with compiler source file/line/function and marks them "internal error". A user sees:

```
initzero.silly:1:9: internal error: BisonParseListener.cpp:607:emitParseError: syntax error, unexpected RIGHT_CURLY_BRACKET_TOKEN
```

Fix — one line change:

```cpp
    emitUserError( loc, msg, currentFuncName );
```

**Estimated effort:** 5 minutes. **Impact:** High — every Bison parse error currently looks like a compiler bug.

---

### 2. `generateCall` Does Not Bounds-Check `funcType.getInputs()`

```cpp
for ( const auto& e : args )
{
    mlir::Type ty = funcType.getInputs()[i];  // unchecked index
    ...
    i++;
}
```

If a call site passes the wrong number of arguments, this indexes out of bounds. Both frontends are affected since this code now lives in `Builder`. A pre-loop check:

```cpp
if ( args.size() != static_cast<size_t>( funcType.getInputs().size() ) )
{
    emitUserError( loc, llvm::formatv( "call to '{0}': expected {1} arguments, got {2}",
        name, funcType.getInputs().size(), args.size() ).str(), currentFuncName );
    return {};
}
```

**Estimated effort:** 30 minutes. **Impact:** Prevents out-of-bounds crash on wrong-arity calls.

### 3. `getLocation` Silently Drops End Location

```cpp
mlir::Location BisonParseListener::getLocation( const silly::BisonParser::location_type& bLoc )
{
    return mlir::FileLineColLoc::get( builder.getStringAttr( sourceFile ),
                                      bLoc.begin.line, bLoc.begin.column );
}
```

Every expression in the `Expr` tree gets the statement's start location — no sub-expression has its own location. This is noted in `TODO.md` but flagged here explicitly as it affects all expression-level error messages in the Bison frontend.

## 💡 OBSERVATIONS

### What's Going Really Well ✅

The Bison frontend went from zero to 174/248 tests passing in 14 days. The `Builder` factoring is the architectural reason this was possible — `BisonParseListener` is only 804 lines because it delegates nearly everything. The monolithic `ParseListener.cpp` (2,369 lines) is gone, cleanly decomposed into three independently comprehensible components.

The `-fno-rtti` auto-detection in `bin/build` is a real usability improvement — default LLVM builds previously made the ANTLR4 frontend unusable without a custom LLVM build.

The test restructuring is complete. One `tests/` directory, one `lit.cfg.py`. Generated tests are now self-documenting.

### Areas of Concern 📈

The `emitParseError` → `emitInternalError` routing means every syntax error in the Bison frontend currently presents as a compiler bug. This is a five-minute fix that should happen before anything else.

The raw `new` in `SourceManager::createCU` has now appeared in three consecutive reviews. It is not blocking anything, but it signals accumulated technical debt in the ownership model that will eventually matter.

The 74 remaining Bison test failures (30% of the suite) are the next major milestone. Based on the progression pattern, many are likely attributable to location info gaps or small missing features rather than fundamental issues.

### What Not to Do ⚠️

Don't start WHILE/BREAK/CONTINUE or multi-dimensional arrays until the Bison frontend triage is complete and the `emitParseError` severity issue is fixed. Don't let the IMPORT correctness gaps (cycle detection, signature mismatch) drift further — the feature is being used and the gaps are real correctness holes.
